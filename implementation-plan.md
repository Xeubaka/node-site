# Horus Awards — Implementation Plan

A stone-design competition platform where users submit their stone/marble designs,
browse competitors, and compete in awards. Modeled on the provided landing-page
mockup ("THE BEST STONES DESIGN" / "HORUS AWARDS 2025").

---

## 1. Goals & Scope

Build a small, container-first web application with a clear split between a
static/MPA frontend and a vanilla Node.js JSON API, backed by a simple SQL
database, fronted by nginx.

**In scope (v1)**
- Public landing page matching the mockup (hero, 3-step band, featured award
  banner, competitors gallery, footer).
- Browse competitors gallery with "See more" pagination.
- Submission detail view.
- User auth: register, log in, log out (JWT).
- Authenticated design submission (title, description, image, category).
- **Admin moderation dashboard** (admin-only): review the submission queue,
  approve/reject, manage competitions.
- **Voting (5-star)** for **all logged-in users**, including admins. One rating
  per user per submission (updatable). Admin ratings double as the judging score
  and are surfaced separately from the public average.
- **Comments** on submissions for all logged-in users. Comments authored by an
  admin display an **"admin" label**.
- **Real-time notification system (WebSocket)**: whenever an admin changes a
  user's submission status, an admin rates the submission (judging), or anyone
  comments on it, the owner receives a live in-app notification pushed over a
  WebSocket. Notifications are also persisted so history/backfill survives
  reconnects.
- Static content pages: Rules, Regulations, FAQ, Privacy & Legal, Policies.
- Automated tests with coverage on both frontend and backend.
- Everything runs via `docker compose`, routed through nginx.
- **Architecture graph** generated by `graphify` (dev/CI tool) — see §11.

**Out of scope (v1, noted for later)**
- Payments and email delivery (explicitly excluded).
- Multi-criteria judging rubrics (v1 judging = an admin's 5-star rating).
- Threaded/nested comment replies (flat comments only in v1).

## 2. Tech Stack

| Layer     | Choice                                                        |
|-----------|--------------------------------------------------------------|
| Frontend  | Static HTML + MaterializeCSS, vanilla JS + jQuery. No SPA framework, no bundler required (optional light tooling only). |
| Backend   | Node.js standard library only (`http`, `url`, `crypto`, etc.). No Express/Fastify. Thin hand-rolled router. |
| Database  | **SQLite via the built-in `node:sqlite` module (Node 22+).** Simple SQL, file-based, **zero runtime dependencies.** |
| Proxy     | nginx — serves static frontend + reverse-proxies `/api/*` to backend. |
| Runtime   | Docker + docker compose. |
| Tests     | Backend: `node:test` + `c8` coverage. Frontend: Jest + jsdom (or `node:test` + jsdom) for unit; Playwright optional for e2e. |

> **Zero-dependency stack (decided):** the backend uses Node's built-in
> `node:sqlite` module, so the application has **no runtime npm dependencies** —
> both the server layer and the database client are standard library. The DB is
> a single file on a mounted volume; no separate database container is needed.
> Requires a Node 22+ base image. All DB access stays behind `db/index.js` so the
> engine could be swapped later if desired.

## 3. Architecture

```
                    ┌─────────────────────────────┐
   Browser  ──────► │            nginx            │
                    │  :80                        │
                    │  /            → frontend     │  (static files)
                    │  /api/*       → backend:3000 │  (reverse proxy)
                    └───────┬─────────────┬────────┘
                            │             │
                    ┌───────▼──────┐  ┌───▼─────────────┐
                    │  frontend    │  │   backend       │
                    │  static vol  │  │   Node http     │
                    │  (nginx)     │  │   :3000         │
                    └──────────────┘  └───┬─────────────┘
                                          │ node:sqlite
                                    ┌─────▼──────────────┐
                                    │ SQLite file        │
                                    │ (mounted volume)   │
                                    └────────────────────┘
```

- Frontend is served by nginx directly from a shared volume / image layer.
- All API traffic is same-origin under `/api/*`, so no CORS needed in prod.
- **WebSocket**: `/api/ws` is proxied by nginx with `Upgrade` / `Connection`
  headers and `proxy_http_version 1.1` so the connection upgrades to the backend.
- Backend never exposed publicly; only nginx binds a host port.
- Uploaded images stored on a mounted volume (`/uploads`), served by nginx
  under `/uploads/*`.

## 4. Data Model (initial)

```sql
users         (id, email UNIQUE, password_hash, display_name,
               role, created_at)               -- role: 'user' | 'admin'
competitions  (id, slug UNIQUE, title, year, description, is_active, created_at)
submissions   (id, user_id FK, competition_id FK, title, description,
               category, image_path, status, created_at)
                 -- status: pending | under_review | approved | rejected
ratings       (id, submission_id FK, user_id FK, stars, created_at, updated_at)
                 -- stars 1..5; UNIQUE(submission_id, user_id); upsert per user
comments      (id, submission_id FK, user_id FK, body, created_at)
                 -- flat; "admin" label derived from the author's role
notifications (id, user_id FK, type, title, message, submission_id FK,
               is_read, created_at)
                 -- type: status_change | judged | comment
```

- **Auth is stateless (JWT)** — no sessions table. Tokens are signed with
  `SESSION_SECRET` using `crypto` (HMAC-SHA256), carry `sub` (user id) + `role`,
  and are verified on each request.
- **Roles**: `role` on `users` gates admin endpoints. Seed one admin account
  from env (`ADMIN_EMAIL` / `ADMIN_PASSWORD`) on first boot.
- Passwords hashed with `crypto.scrypt` (built-in) + per-user salt.
- **Submission lifecycle**: `pending` → (admin) `under_review` → `approved` |
  `rejected`. Only `approved` submissions appear in the public gallery.
- **Voting (5-star)**: any logged-in user may rate a submission 1–5 stars; a
  second rating from the same user updates their existing row (upsert). The
  submission detail shows the **public average** (ratings by non-admins) and the
  **judge score** (average of admin ratings) separately — both derived by
  joining `ratings.user_id → users.role`, so no extra column is needed.
- **Judging = admin rating**: when an admin creates/updates a rating, it counts
  as the judging score **and** notifies the owner (`type=judged`). Non-admin
  votes update the public average but do **not** create notifications (avoids
  spam).
- **Comments**: any logged-in user may comment (flat, no threads). The "admin"
  label is derived from the author's current role at render time. A new comment
  notifies the submission owner (`type=comment`) unless the author is the owner.
- **Notifications** are created server-side inside the same transaction as the
  triggering action, addressed to the submission's owner, **and** pushed live
  over WebSocket to any connected socket for that user (see §10). Persisted in
  the DB for history/backfill. In-app only (no email).
- Seed data: one active competition ("Horus Awards 2025"), one admin user, and a
  handful of demo submissions so the gallery renders on first boot.

## 5. API Surface (v1)

All JSON, prefixed `/api`. Auth via `Authorization: Bearer <token>` or session cookie.

Auth levels: **no** = public, **user** = any logged-in user, **admin** = `role=admin`.

| Method | Path                                   | Auth  | Purpose                             |
|--------|----------------------------------------|-------|-------------------------------------|
| POST   | /api/auth/register                     | no    | Create account (role=user)          |
| POST   | /api/auth/login                        | no    | Log in, return JWT                  |
| POST   | /api/auth/logout                       | user  | Client-side token discard (no-op)   |
| GET    | /api/auth/me                           | user  | Current user + role                 |
| GET    | /api/competitions                      | no    | List competitions (`?open=1` = accepting submissions) |
| GET    | /api/awards                            | no    | All competitions + winner + contestants (awards carousel) |
| GET    | /api/competitions/:slug                | no    | Competition detail (+ window, winner, isOpen) |
| GET    | /api/submissions                       | no    | List approved (paginated)           |
| GET    | /api/submissions/:id                   | no    | Submission detail (+ scores if own/admin) |
| POST   | /api/submissions                       | user  | Create submission (multipart, competitionId) |
| GET    | /api/submissions/mine                  | user  | Caller's own submissions (any status) |
| GET    | /api/submissions/:id/ratings           | no    | Public avg + judge score + my vote  |
| PUT    | /api/submissions/:id/ratings           | user  | Cast/update my 5-star vote (upsert)  |
| GET    | /api/submissions/:id/comments          | no    | List comments (with admin flag)     |
| POST   | /api/submissions/:id/comments          | user  | Add a comment → notifies owner      |
| GET    | /api/notifications                     | user  | Own notifications (?unread=1)       |
| PATCH  | /api/notifications/:id/read            | user  | Mark one read                       |
| POST   | /api/notifications/read-all            | user  | Mark all read                       |
| GET    | /api/ws                                | user* | WebSocket upgrade for live notifs    |
| GET    | /api/admin/submissions                 | admin | Moderation queue (?status=)         |
| PATCH  | /api/admin/submissions/:id             | admin | Change status → notifies owner      |
| GET    | /api/admin/competitions                | admin | Competitions + stats (most-voted, winner) |
| POST   | /api/admin/competitions                | admin | Create competition (+ window)       |
| PATCH  | /api/admin/competitions/:id            | admin | Update competition (window, active) |
| POST   | /api/admin/competitions/:id/award      | admin | Award a winner (only after submissions close) → notifies owner |
| GET    | /api/health                            | no    | Liveness for compose healthcheck    |

`user*` on `/api/ws`: the socket authenticates via a first `{type:"auth",token}`
message (not a URL query param — tokens must never appear in URLs).

Admin judging is **not** a separate endpoint: an admin simply uses
`PUT /api/submissions/:id/ratings` like any user, and the server treats an
admin's rating as the judge score and notifies the owner.

Standard error envelope: `{ "error": { "code", "message" } }`.
Non-admin hitting an `/api/admin/*` route → `403` with the envelope.

guard is UX only; the API is the real boundary.

## 8. Testing Strategy

- **Backend unit/integration**: `node:test`; spin the HTTP server against an
  in-memory SQLite database, assert on routes, auth, validation, and the error
  envelope. Coverage via Node's built-in `--experimental-test-coverage`.
- **Frontend unit**: UMD logic modules (`validators`, `util`, `gallery`,
  `rating`, `comments`, `auth`, `api`, `ws`) tested with `node:test` — `fetch`
  and `WebSocket` are mocked. No Jest/jsdom, no install.
- **e2e (optional)**: Playwright hits nginx and walks the full loop:
  register → submit → admin approves → admin scores → user sees notification.
- **Admin/role coverage**: explicit tests that non-admins get `403` on every
  `/api/admin/*` route, and that status/score changes create exactly one
  notification for the correct owner.
- **Coverage target**: 80% lines/branches on backend business logic; frontend
  logic modules 70%+. HTML/markup excluded.
- CI runs `docker compose` build + backend tests + frontend tests on every push.

## 9. Decisions (locked)

- **Auth mechanism**: **JWT bearer tokens** — stateless, signed with
  `SESSION_SECRET` (HMAC-SHA256 via `crypto`). Token sent in
  `Authorization: Bearer <token>`; frontend stores it in memory/localStorage.
  No sessions table. Note the XSS surface of token-in-localStorage — the
  frontend must escape all user-provided text (see `frontend/CLAUDE.md`).
- **Database**: **SQLite via built-in `node:sqlite` (Node 22+)** — zero runtime
  dependencies, file-based on a mounted volume, no DB container.
- **Frontend build**: **none** — raw files served by nginx; keeps the
  "no framework" constraint honest.
- **Notification transport**: **WebSocket** (`/api/ws`), real-time, persisted in
  DB for backfill. Browser uses the native `WebSocket` API (no client lib).
- **WebSocket server**: **hand-rolled minimal RFC-6455** on Node's `http` upgrade
  event using built-in `crypto` — preserves the backend's **zero runtime
  dependencies**. *Trade-off:* more code to write and test than pulling in `ws`.
  If the hand-rolled layer proves fiddly, `ws` is the sanctioned fallback (it
  would be the app's only runtime dependency). Kept behind `lib/ws.js` so the
  swap is localized.
- **Voting**: **5-star ratings for all logged-in users**; an admin's rating is
  the judge score. Replaces the earlier admin-only multi-criteria rubric.

See `frontend/CLAUDE.md` and `backend/CLAUDE.md` for per-package conventions.

## 10. Admin, Voting, Comments & Real-time Notifications

**Admin/moderation dashboard** (`admin/index.html`, `/api/admin/*`)
- Only `role=admin` users; enforced server-side on every admin route (403 otherwise).
- Moderation queue: filter submissions by status; approve → `approved`,
  reject → `rejected`, or move to `under_review`. Each change writes a
  notification to the submission owner in the same DB transaction.
- Competition management: create/edit competitions, toggle `is_active`.

**Voting — 5-star, all users** (`PUT /api/submissions/:id/ratings`, `rating.js`)
- Any logged-in user (admin or not) casts a 1–5 star rating; re-rating upserts
  their single row. Anonymous visitors see the averages but can't vote.
- `submission.html` shows the **public average** (non-admin ratings) and the
  **judge score** (admin ratings) separately — both derived by joining to
  `users.role`; no extra column.
- **Judging = an admin's rating.** When an admin rates a submission, the server
  notifies the owner (`type=judged`). Non-admin votes do not notify (anti-spam).

**Comments — all users, admin label** (`/api/submissions/:id/comments`, `comments.js`)
- Any logged-in user posts a flat comment. The list renders an **"admin" badge**
  next to comments whose author currently has `role=admin`.
- A new comment notifies the submission owner (`type=comment`) unless the author
  is the owner. Escape comment bodies on render (XSS).

**Real-time notifications — WebSocket** (`GET /api/ws`, `notifications.js`, `notifications.html`)
- **Transport**: a WebSocket at `/api/ws`, proxied by nginx (with the
  `Upgrade`/`Connection` headers). The browser uses the native `WebSocket` API —
  no client library.
- **Auth**: the client opens the socket, then sends `{type:"auth",token}` as the
  first message; the server verifies the JWT and registers the socket under the
  user id (`Map<userId, Set<socket>>`). Tokens never go in the URL.
- **Delivery**: notifications are still written to the DB (source of truth) inside
  the triggering transaction; immediately after commit the server pushes the new
  notification JSON to that user's live sockets. The navbar badge and any open
  notifications page update in real time.
- **Backfill / resilience**: on (re)connect the client calls
  `GET /api/notifications?unread=1` to catch anything missed while disconnected.
  The client auto-reconnects with backoff. So correctness never depends on the
  socket staying up — WS is the fast path, REST + DB is the guarantee.
- **Server implementation**: see §9 decision — a minimal hand-rolled RFC-6455
  layer (handshake via `crypto` SHA-1 accept key; text-frame parse/encode; ping/
  pong; close) to preserve the zero-runtime-dependency property. `lib/ws.js`
  owns this; routes call `ws.pushToUser(userId, payload)`.
- **Tests**: notification rows created with the correct owner and exactly once
  per triggering action (no self-notification); `lib/ws.js` frame encode/decode
  and the accept-key handshake unit-tested; an integration test asserts a
  connected client receives a pushed notification after an admin action.
